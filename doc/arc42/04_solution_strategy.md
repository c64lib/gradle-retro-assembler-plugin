# 4. Solution Strategy

This section summarizes the key architectural decisions that satisfy the quality goals in [§1.2](01_introduction_and_goals.md#12-quality-goals) and the constraints in [§2](02_architecture_constraints.md). Each is expanded further in [§8 Crosscutting Concepts](08_crosscutting_concepts.md) and recorded formally in [§9 Architecture Decisions](09_architecture_decisions.md).

| Strategy | Addresses | Summary |
|----------|-----------|---------|
| **Hexagonal architecture per domain** | Extensibility, maintainer/agent legibility | Every bounded context (`compilers`, `dependencies`, `emulators`, `testing`, `processors`, `crunchers`, `flows`) is internally split into `src/{domain,usecase}` (business logic) and `adapters/{in,out}/<technology>` (technology-specific glue). Adding a new assembler dialect or asset processor means adding a new domain module that follows this shape, not modifying existing ones. |
| **Gradle isolated as an adapter concern** | Extensibility, testability | Gradle's `Project`, `Task`, and extension APIs never appear in domain/use-case code — they are confined to `adapters/in/gradle` (DSL, tasks) and `adapters/out/gradle` (file/process operations via Gradle APIs). This lets use cases be unit-tested without spinning up a Gradle project, and means a future non-Gradle front end (unlikely, but architecturally possible) would only require new adapters. |
| **Use-case-per-operation with a single `apply` method** | Maintainer/agent legibility | Every unit of business behavior is a `*UseCase.kt` class with one public `apply(payload)` method (e.g. `KickAssembleUseCase`, `ProcessCharpadUseCase`). This keeps each use case independently testable and gives contributors and AI agents one predictable entry point per capability, rather than multi-method service classes with implicit call ordering. |
| **Manual dependency injection in the plugin entry point** | Extensibility, build correctness | `RetroAssemblerPlugin.apply()` (`infra/gradle/src/main/kotlin/com/github/c64lib/gradle/RetroAssemblerPlugin.kt`) is the single place where use cases are constructed and wired to their port implementations and Gradle tasks, inside `project.afterEvaluate`. There is no DI framework (Spring, Dagger, Koin) — wiring is explicit constructor calls, keeping the dependency graph visible and grep-able rather than hidden behind annotations/reflection. |
| **`flows` as an orchestrating bounded context, not a new processor** | Extensibility, incremental build performance | The `flows` domain does not reimplement compilation/processing logic; it defines a step-based Pipeline DSL and delegates actual work to the same use cases as the direct tasks, through its own ports (`flows/src/main/kotlin/.../domain/port/`, e.g. `AssemblyPort`, `CharpadPort`, `ExomizerPort`). This means `flows` and the direct (`asm`, `charpad`, …) task paths share one implementation of each capability. |
| **Gradle Workers API for parallelism; file-based task dependencies for incrementality** | Incremental build performance | No custom threading is used anywhere in the codebase (mandated by `CLAUDE.md`). Parallel execution of independent flow steps/flows is achieved by having `FlowTasksGenerator` derive Gradle task `dependsOn`/input-output relationships from the domain's `FlowDependencyGraph` and each step's declared `inputs`/`outputs`, so Gradle's own scheduler (under `--parallel`) and up-to-date checking do the work — see [PLAN-0001](../../plans/PLAN-0001_pipeline-dsl-parallel-execution.md) for the design history. |
| **New domains must be registered as `compileOnly` in `infra/gradle`** | Build correctness | Because DI is manual and module inclusion is explicit (`settings.gradle.kts`), a new domain module is inert until `infra/gradle` depends on it and `RetroAssemblerPlugin` wires its use case(s) to a task. This is a deliberate, enforced two-step (module + wiring) rather than auto-discovery, trading a small amount of ceremony for an explicit, auditable task graph. |
| **Markdown + Mermaid for architecture documentation** | Maintainer/agent legibility | This arc42 set itself is Markdown with Mermaid diagrams under `doc/arc42/`, chosen so it renders natively on GitHub with no build toolchain, stays diffable, and can be kept current by both humans and agents editing text — see [§9](09_architecture_decisions.md) for the decision record. |
